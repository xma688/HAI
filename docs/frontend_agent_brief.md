# HAI 前端重设计说明（交给设计 / 前端 Agent）

## 一句话产品描述

HAI 是一个面向中文对话场景的情感型 AI 虚拟陪伴 Demo：用户输入文字或语音后，系统在同一轮推理中生成自然语言回复，并输出情绪、表情、动作、语气、语速和强度等结构化控制信息，再由 TTS 合成语音并驱动 Live2D 角色完成表情、动作、口型与声音的协同呈现。

## 可直接复制给 Agent 的项目描述

请为 HAI 设计并实现一套全新的响应式前端，不要复制或小修当前 Gradio 页面。HAI 不是普通聊天机器人，也不是单纯的 Live2D 展示页；它的核心体验是“我说一句话，角色理解我的情绪，以合适的内容、语气、表情和动作回应我”。前端应让用户首先感受到一个正在倾听和回应的角色，而不是一堆模型参数、状态卡片和技术模块。桌面端以对话与实时角色组成一个完整舞台，移动端保持角色完整可见且输入始终易用。技术状态、延迟、emotion / expression / gesture 等字段属于可展开的诊断信息，不应抢占主视觉。项目已有一套深靛蓝、青色、淡紫和暖珊瑚色的品牌图片，可用于首次进入、空状态或页面下方的产品叙事，但不要把全部图片堆在首屏。设计需要覆盖空状态、输入、处理中、文字回复完成、语音生成中、角色表演中、完成、警告和失败状态，并为未来的流式文字与分阶段事件预留表现方式。

## 产品目标与用户任务

- 主要用户：希望体验“更有情绪表达的 AI 对话”的课程评审者、开发者和普通体验者。
- 第一任务：快速输入一句话并获得自然、连续的回复。
- 第二任务：看到角色的表情、动作和口型与回复语义保持一致。
- 第三任务：听到可理解的中文语音；语音失败时仍能正常阅读文字，不播放占位哔声。
- 辅助任务：需要时查看本轮的情绪、表情、动作、语气、强度、延迟和降级警告。

## 当前真实能力与边界

一轮输入会经过以下链路：

```text
文本 / 录音
  → 可选语音转文字
  → 会话上下文与用户画像
  → LLM 一次生成 reply_text + emotion + expression + gestures + voice 参数
  → 解析、枚举校验、一致性规则与动作冷却
  → TTS 生成 WAV
  → Live2D bridge 更新角色状态、播放音频、驱动口型 / 表情 / 动作
  → 返回文字、音频、角色控制信息、延迟和警告
```

当前接口是“整轮完成后返回”，不是真正的 token streaming。前端可以明确展示“理解 → 回复 → 声音 → 表演”的进度感，但不能伪装成已有流式接口。设计和组件状态应方便以后接入增量文字与阶段事件。

主要输出数据：

- `reply_text`：角色的文本回复。
- `avatar_command.emotion`：neutral / happy / supportive / thoughtful / confused / surprised / serious / apologetic。
- `avatar_command.expression`：neutral / smile / soft_smile / thinking / confused / surprised / concerned / serious。
- `avatar_command.gestures`：idle / nod / wave / head_tilt / think / explain / agree / small_bow，最多两个。
- `voice_style`、`speaking_rate`、`gesture_intensity`、`pause_before_speech_ms`。
- `audio_path`、分阶段 latency、warnings。

## 信息架构建议

不要沿用“营销首页 + Demo 面板 + 图片图库 + 功能卡片全部同权”的拼接结构。建议将页面收敛为三个层级：

1. 精简顶栏：品牌、连接状态、必要设置；提供商名称放入设置或诊断区。
2. 核心舞台：一个统一容器内呈现 Live2D 角色、对话历史和输入栏。角色回复只出现一次，不要在聊天记录和角色侧重复显示同一大段文本。
3. 次级内容：可折叠的“本轮表现 / 技术诊断”，以及按需出现的产品故事或首次进入说明。

首次进入可用 Hero 图片和一句简短引导建立氛围；用户开始对话后，界面应自然转入沉浸式会话模式，让营销内容退出主舞台。

## Live2D 舞台硬约束

- 角色必须采用 `contain` 思路按模型原始尺寸缩放，不能根据已经缩放过的宽高反复计算。
- 默认至少保留约 12% 的四周安全空间；头顶、双手和身体下沿不可被裁切。
- 桌面端角色舞台建议占核心舞台宽度的 40%–46%，高约 560–680px，角色视觉中心略高于容器几何中心。
- 窄屏时优先保证完整角色，而不是填满容器；角色舞台建议为 34–42vh，并设置合理的最小 / 最大高度。
- 角色区加载中、模型加载失败、bridge 断开时要有克制的占位和重试提示，不能只显示空白 iframe。
- 角色的 idle 动效应轻微；支持 `prefers-reduced-motion`，避免容器整体晃动造成眩晕。

## 响应式与可用性

- ≥ 1200px：对话与角色并列，构成一个整体舞台；输入栏固定在对话区域底部。
- 768–1199px：允许角色在上、对话在下，或使用紧凑双栏，但不得出现横向滚动。
- < 768px：角色保持完整可见，聊天记录获得主要纵向空间，输入栏接近视口底部且不被软键盘遮挡。
- 所有主要操作支持键盘；发送键、录音状态、停止录音、重试和清空对话有明确标签。
- 文本对比度满足基本可读性；不要用低对比度灰字承载关键状态。

## 必须设计的交互状态

- 首次进入 / 空对话。
- 输入聚焦、可发送、空输入、录音中、转写中、转写失败。
- 请求排队、LLM 处理中、TTS 处理中、角色准备中。
- 文字已返回但语音尚未完成。
- 角色说话 / 表演中、播放完成、用户手动重播。
- LLM 失败、TTS 失败、Live2D 加载失败、bridge 离线、部分降级成功。
- 清空确认与清空后的状态。
- 长文本、连续多轮、超长中文或英文、无音频返回。

错误不应清掉用户刚刚输入的内容。TTS 或角色失败时保留文字回复，并提供局部重试；不要用模拟哔声伪装真实语音。

## 视觉方向与已有资产

视觉关键词：安静、可信、温暖、夜间陪伴、克制的数字感。基础色为深海军蓝 / 靛蓝，强调色为青色与淡紫，暖珊瑚色只用于人情味和关键反馈。减少玻璃卡片、发光描边、徽章、渐变和大段营销文案的同时出现，避免“所有东西都很高级但没有主次”的效果。

已有资源位于 `assets/brand/`：

- `hai-companion-icon.png`：品牌图标，可用于 favicon、顶栏和助手头像。
- `hero-companion.jpg`：首次进入或空状态主视觉。
- `emotion-understanding.jpg`：情绪理解叙事图。
- `natural-voice.jpg`：自然语音叙事图。
- `expressive-avatar.jpg`：角色表情与动作叙事图。
- `quiet-companionship.jpg`：安静陪伴叙事图。

这 5 张照片属于同一套视觉语言，但不要求同屏全部展示。优先把它们用于空状态、引导、分段背景或页面下方叙事；核心会话阶段的视觉主角必须是 Live2D 角色。

## 技术接入约束

- 当前前端由 Python Gradio Blocks 生成，主入口为 `src/hai_avatar/ui/gradio_app.py`，样式为 `src/hai_avatar/ui/styles.css`。
- Live2D 通过本地 bridge iframe 嵌入；默认主页面 `127.0.0.1:7860`，bridge `127.0.0.1:7861`。
- 不改变后端 pipeline、数据模型和 API 语义；若改为独立前端，先为现有 `respond`、clear、audio / ASR 能力建立稳定接口适配层。
- 保留真实模式与 Mock 模式；界面可以区分“真实服务”“演示数据”，但不要让提供商徽章主导页面。
- 静态资源必须从项目内可靠加载，避免在前端中写死个人绝对路径。

## 验收标准

- 用户在 5 秒内能判断在哪里输入、角色在哪里、当前是否可用。
- 桌面和 390px 宽移动视口均无横向滚动，Live2D 全身完整可见。
- 一轮真实对话可完成：输入 → 自然文字回复 → 中文语音 → 角色表情 / 动作 / 口型。
- 回复不重复占据两个同级区域；诊断字段默认不抢主视觉。
- 页面加载、等待、局部失败和完全失败均有明确状态。
- 使用已有品牌资产后仍保持单一视觉中心，不形成图库和功能卡片的拼贴感。
- 设计稿需同时给出桌面、移动端和关键状态，不只给静态首页。
